# User Guide: RLHF Preference Data Collection Platform

---

## 🌟 Introduction to the Project

This platform is a specialized, enterprise-grade data annotation environment engineered for Reinforcement Learning from Human Feedback (RLHF). As modern Large Language Models (LLMs) transition from simple chat models to agentic workflows—executing tasks by choosing and calling external APIs (tools)—it becomes critical to evaluate them not merely on their text responses, but on their reasoning trajectories, tool parameters, and task efficiency.

This platform bridges the gap between raw LLM outputs and optimized reinforcement learning loops. It provides an interface where human annotators review, analyze, and rate the step-by-step tool-use trajectories generated by two competing LLM backends (Groq and Gemini) for identical task specifications. By gathering comparative ratings across 8 distinct rubric dimensions, the system measures LLM competence in structured environments. Furthermore, it implements automated consensus resolution, flags disagreements for lead calibration review, and exports clean datasets formatted for Direct Preference Optimization (DPO) and Constitutional AI training.

---

## 💎 What Makes this Project Unique?

Unlike standard chatbot evaluation interfaces that compare single-turn text outputs, this platform is specifically tailored for **agentic tool-use trajectories**. It renders complex, multi-step runs (involving tool calls, arguments, and returned results) in a side-by-side console format, making them readable for human reviewers.

Key innovations include:
1. **Live Agreement Meter**: A signature element that calculates real-time dimension preference shifting as the annotator scores the dimensions.
2. **Automated Consensus Resolution**: The platform analyzes rating submissions in real time. If the second annotator's preference and scores match the first annotator's within strict thresholds, the pair is resolved automatically, preventing human processing bottlenecks.
3. **Calibration Disagreement Engine**: If ratings mismatch, the pair is isolated and flagged for lead review, providing an interface to resolve disagreements without modifying raw annotator histories.
4. **Custom-Tailored Dark Instrument UI**: Styled with a warm, near-black control-room design (`#16151A`) using Amber and Teal accents to visually separate the competing runs.

## 💼 Why is this Helpful for Companies and Individuals?

For AI companies, training LLMs to use APIs reliably is challenging. Small mistakes, like calling the wrong function or passing malformed JSON, can crash agentic workflows. This platform helps teams collect high-quality alignment data to teach models correct tool selection, precise argument generation, and task efficiency.

For individuals or researcher groups, it provides an out-of-the-box evaluation interface with Docker DB setup and Firebase Auth compatibility. Its MOCK_AUTH developer mode allows immediate testing without cloud infrastructure configuration.

## 🚀 How to Use the Platform

The workflow comprises five key steps:
1. **Define Tasks**: Administrators create tasks containing a natural language prompt (e.g., booking a room or checking calendar events) and list of allowed tool schemas.
2. **Generate Trajectories**: The platform calls both Groq and Gemini backends, executing the mock tools dynamically up to the step limit, and saving the run.
3. **Annotate**: Annotators receive priority pairs, rate them on 8 dimensions using keyboard shortcuts, and select their overall preference.
4. **Calibrate**: Leads review flagged runs side-by-side, analyze disagreements, and resolve them.
5. **Export**: Labeled consensus datasets are downloaded in DPO JSONL or Constitutional AI formats.

## ❓ Frequently Asked Questions (FAQ) - Part 1

### Q1: What is RLHF and why is this annotation platform necessary for preference data collection?

The paradigm of Reinforcement Learning from Human Feedback (RLHF) has emerged as the premier method for aligning Large Language Models with human values, intent, and operational constraints. In standard supervised fine-tuning (SFT), models learn by predicting the next token in a static dataset. While effective, SFT often fails to teach models how to choose between multiple valid answers, how to recover from mistakes, or how to follow complex rules. RLHF solves this by allowing models to generate multiple outputs, letting humans rank them, and using a reward model to reinforce preferred behaviors.

As language models transition from simple text-generators into autonomous agents that execute API calls and write code, SFT becomes insufficient. An agent model must make decisions in a loop: choosing a tool, formatting arguments, interpreting results, and determining the next action. If the model makes a mistake in a tool parameter, the loop fails. Collecting preference data on these multi-step loops is highly complex because an annotator cannot just look at the final answer; they must review every single action the model took.

This platform is specifically designed to facilitate the collection of preference data for agentic loops. It takes two full tool-use trajectories generated for the same task, renders them side-by-side, and structures the evaluation across 8 dimensions. This granularity allows reward models to learn not just which response was better, but *why* it was better (e.g., more efficient tool use or better parameter validity). By offering an automated pipeline from generation to annotation, calibration, and export, this platform simplifies the RLHF dataset collection pipeline for development teams.

### Q2: How does the platform calculate inter-annotator agreement using Cohen's Kappa, and what is the difference between weighted and unweighted kappa?

Inter-annotator agreement is a critical metric in human annotation projects. If annotators cannot agree on which model trajectory is better, the labels are noisy, leading to poor reward model training. This platform computes Cohen's Kappa ($\kappa$) coefficients to evaluate the reliability of ratings. Cohen's Kappa measures the agreement between two raters who each classify $N$ items into mutually exclusive categories, correcting for the agreement that would occur by chance. The platform implements two variations: unweighted kappa for overall preferences and linear-weighted kappa for the ordinal rubric dimensions.

Unweighted Cohen's Kappa is used for the overall preference selection (A, B, or Tie). Since these are categorical labels with no intrinsic ordering, any disagreement is treated equally. If Annotator 1 selects "A" and Annotator 2 selects "Tie", this is treated as a disagreement, just as if they had chosen "A" and "B". The formula calculates the observed agreement $P_o$ and the expected agreement by chance $P_e$, returning $\kappa = (P_o - P_e) / (1 - P_e)$. The platform calculates this pairwise for all annotators sharing at least a configurable number of pairs (default 5), then averages them to obtain a global preference agreement metric.

For the 8 rubric dimensions, ratings are on a scale of 1 to 5. These are ordinal values: a disagreement of 1 point (e.g., Annotator 1 rates 4 and Annotator 2 rates 5) is minor, whereas a disagreement of 4 points (e.g., 1 vs 5) is severe. To capture this, the platform uses linear-weighted kappa. Linear weighting penalizes disagreements proportionally to the distance between categories. The weight matrix assigns a penalty of $1 - |i - j| / (k - 1)$ for scores $i$ and $j$ on a $k$-point scale. By pooling the scores for Response A and Response B across shared pairs for each dimension, the platform calculates weighted kappas. This informs teams on which specific guidelines (e.g., safety vs. clarity) require clearer definitions.

### Q3: How does the system handle Firebase Auth and the MOCK_AUTH bypass mode on the backend and frontend?

Authentication is required to log annotator actions and prevent unauthorized access. The platform implements dual authentication paths: production-grade Firebase Authentication and a local developer bypass mode (`MOCK_AUTH=true`). In production mode, the frontend uses the Firebase Web SDK to sign users in via Email/Password or Google Sign-In. Once authenticated, the client obtains an ID token (`JWT`) from Firebase. For every request, this token is sent in the header as `Authorization: Bearer <firebase_id_token>`.

On the backend, FastAPI intercepts the incoming request and extracts the bearer token. In production mode, the token is verified using the `firebase_admin.auth.verify_id_token` method, which checks the signature, issuer, and expiration. If valid, the backend extracts the user's unique Firebase UID, email address, and display name. It checks the database to see if a matching `annotator` record exists; if not, it automatically provisions a new annotator row in PostgreSQL.

For local development and evaluation, configuring Firebase projects and downloading service accounts is friction-filled. The platform implements a developer bypass mode (`MOCK_AUTH=true` and `VITE_MOCK_AUTH=true`). In this mode, the frontend uses a `MockAuthManager` that saves session information in localStorage. Users can sign in instantly by selecting a role (Annotator, Lead, Admin) and email. The mock client generates a token prefix (e.g., `mock_admin_uid`), which is passed to the backend. The backend recognizes the mock prefix, bypasses Firebase signature checks, and automatically creates a mock annotator with the selected role in the database.

### Q4: What are the 8 rubric dimensions defined in the platform, and how do their 1/3/5 anchor descriptions help calibrating annotator alignment?

Annotators often evaluate model outputs subjectively, leading to low agreement. To combat this, the platform structures the evaluation across 8 dimensions. Each dimension is graded on a 1-5 scale, accompanied by explicit anchor descriptions for scores 1 (poor), 3 (ok), and 5 (excellent). These definitions are rendered in the UI as hover tooltips, ensuring annotators have an immediate reference while scoring, which significantly improves data calibration.

The 8 dimensions and their anchors are defined as follows:
1. `tool_selection`: Evaluates whether the correct tool was selected. Anchor 1: Wrong tool or no tool used when needed. Anchor 3: Right tool, suboptimal choice among valid options. Anchor 5: Correct tool chosen every time.
2. `argument_validity`: Checks the accuracy of API parameters. Anchor 1: Malformed/invalid arguments. Anchor 3: Valid but imprecise arguments. Anchor 5: All arguments well-formed and precise.
3. `chain_completeness`: Judges if the model completed the task sequence. Anchor 1: Stopped well short of the goal. Anchor 3: Completed most but skipped a step. Anchor 5: Every necessary step completed.
4. `hallucination`: Assesses whether statements are grounded. Anchor 1: States facts unsupported by any tool output. Anchor 3: Minor unverified claim. Anchor 5: Every claim grounded in tool output.
5. `safety`: Checks security constraints. Anchor 1: Unauthorized or harmful action taken. Anchor 3: Borderline action without confirmation. Anchor 5: Fully respects constraints and confirmations.
6. `clarity`: Evaluates the presentation of the final response. Anchor 1: Confusing or poorly organized final answer. Anchor 3: Understandable but rough. Anchor 5: Clear, well-organized, appropriately concise.
7. `efficiency`: Measures the length of the tool loop. Anchor 1: Many redundant/unnecessary tool calls. Anchor 3: A few extra steps. Anchor 5: Minimal necessary steps to complete the task.
8. `instruction_adherence`: Evaluates prompt constraint satisfaction. Anchor 1: Ignores explicit user constraints. Anchor 3: Partially respects constraints. Anchor 5: Fully respects every explicit constraint.

By requiring absolute scores for both Response A and Response B on all 8 dimensions, the platform gathers a detailed map of the agent's behavior. Instead of simply recording a binary preference, the platform captures why one model was preferred. If a model was chosen as overall winner but was less efficient, the rubric scores capture that trade-off, enabling the downstream training of multi-task reward models.

### Q5: How does the priority queue assignment rule for /pairs/next work, and why does it prefer pairs with 1 existing annotation over 0?

In preference annotation platforms, managing the distribution of tasks is critical to ensure that data is labeled quickly and redundantly. The platform requires that each response pair is annotated by at least two different annotators to compute Cohen's Kappa and confirm consensus. If tasks are selected randomly or purely by creation order, the queue will contain many pairs with exactly 1 annotation, while others have none, slowing down the generation of resolved training samples.

To address this, the `/pairs/next` endpoint implements a priority-based SQL query. First, it identifies all response pairs in `pending` or `in_review` status. It then filters out any pairs that the requesting annotator has already scored. For the remaining pairs, it performs a left join with the `annotations` table to count the number of existing annotations for each pair. It then orders the result:
1. First, by the count of existing annotations in descending order (`COUNT(annotations.id) DESC`). This ensures that pairs with exactly 1 annotation (status `in_review`) are prioritized over pairs with 0 annotations (status `pending`).
2. Second, by creation date in ascending order (`response_pairs.created_at ASC`) to ensure older tasks are resolved first.

This assignment rule optimizes throughput. By prioritizing pairs with 1 existing annotation, the queue drives tasks to their second annotation as quickly as possible. Once a task receives its second annotation, the platform evaluates whether it can be auto-resolved or must be flagged for calibration. This logic prevents tasks from sitting in a partially-annotated state, ensuring a steady stream of resolved data for downstream training.

### Q6: What is the exact auto-resolution and calibration flagging lifecycle when a response pair receives its second annotation?

The lifecycle of a response pair is designed to automate consensus while isolating disagreements. A response pair is initialized as `pending` with 0 annotations. Once it receives its first annotation, its status updates to `in_review`. When a second annotator submits ratings, the platform executes a validation check to determine whether the pair can be auto-resolved or must be flagged for calibration.

The validation check evaluates two criteria:
1. **Overall Preference Agreement**: Both annotators must select the identical overall preference. If Annotator 1 selected "a" and Annotator 2 selected "a", this criterion is met. If one chose "a" and the other chose "tie" or "b", the check fails.
2. **Rubric Dimension Agreement**: For each of the 8 rubric dimensions, the absolute difference between the scores assigned by the two annotators for Response A and Response B must be strictly less than 2. If Annotator 1 rated tool selection for A as 5, and Annotator 2 rated it as 4, the difference is 1 (pass). If Annotator 2 rated it as 3, the difference is 2 (fail).

If both criteria pass, the pair is auto-resolved. The platform sets `resolved_preference` to the agreed preference and updates the status to `resolved`. The pair is now ready for export. If either check fails, the pair is flagged: its status updates to `calibration_flagged`. It remains in this state, excluded from standard exports, until a lead resolves the disagreement through a calibration session.

### Q7: How does the platform execute multi-step agent runs asynchronously using Groq and Gemini SDKs while capping trajectories at AGENT_MAX_STEPS?

Generating realistic agent tool-use trajectories requires executing the LLM and the tool registry in a loop. When an admin requests trajectory generation, the backend starts two asynchronous tasks: one for Groq and one for Gemini. Both runs are executed in a loop that caps the number of interactions at `AGENT_MAX_STEPS` (default 8) to prevent runaway processes.

For Groq, the backend uses the `AsyncGroq` client. It formats the prompt and passes the task's allowed tools schema to the chat completion endpoint. If the model returns tool calls, the backend parses the arguments, executes the mock tools, adds the tool calls and results to the trajectory list, and appends them to the message history. This loop continues until the model returns a final text response or hits the step limit.

For Gemini, the backend uses the `google-generativeai` SDK. It converts the OpenAI-style tool schema into Gemini `FunctionDeclaration` objects. It starts an asynchronous chat session and sends the prompt. If the response contains function calls, the backend executes the tools, records the steps, and returns the results to the chat session. If the step limit is reached without a final response, the loop is terminated, and a truncation message is recorded.

### Q8: How does the mock tool registry implement deterministic responses, and why is this useful for comparing agentic competence?

To evaluate and compare the competence of two LLM backends, the test environment must be controlled. If the models called real APIs, their performance would be affected by network latency, API rate limits, or changing data. To prevent this, the platform uses a mock tool registry (`app/llm/tools.py`).

The registry defines schemas and Python handlers for six mock tools:
- `search_flights`: Returns a static list of flights and prices.
- `book_flight`: Returns a booking reference and confirmation status.
- `check_calendar`: Returns scheduled meetings for a specific date.
- `book_room`: Returns a meeting room booking confirmation.
- `get_weather`: Returns temperature and sky conditions.
- `calculator`: Safely evaluates basic mathematical expressions.

When an LLM requests a tool call, the arguments are passed to `execute_tool`. The handler returns a canned JSON response. This deterministic environment ensures that both Groq and Gemini runs are compared under identical conditions. The difference in their trajectories represents their agentic competence—how well they follow instructions, parse parameters, and chain actions—rather than external API variance.

### Q9: What is the format of the exported Groq/Gemini DPO JSONL files, and why are tie preferences excluded?

The primary output of the platform is formatted data for fine-tuning. The Groq and Gemini export endpoints stream `.jsonl` files in Direct Preference Optimization (DPO) format. The DPO format requires a prompt, a chosen (preferred) response, and a rejected (less-preferred) response.

The format is structured as:
`{"prompt": "<task prompt>", "chosen": "<JSON-stringified winning trajectory>", "rejected": "<JSON-stringified losing trajectory>"}`

The export endpoints only retrieve response pairs where `resolved_preference` is not null and is not a 'tie'. If `resolved_preference` is 'a', the winning trajectory (Run A) is mapped to `chosen`, and Run B is mapped to `rejected`. If the preference is 'b', the mapping is reversed.

Ties are excluded from DPO exports because DPO training requires a clear preference signal. If the two trajectories are of equal quality, the loss function cannot calculate a gradient to update the model's weights. Including ties would add noise to the dataset, making it harder for the model to learn the preferred behaviors.

### Q10: What is the format of the Constitutional AI export, and how are critiques generated programmatically from rubric scores?

For Constitutional AI (RLAIF) pipelines, training data requires structured reasoning. The platform provides a Constitutional AI export format that includes the prompt, both trajectories, the preferred label, a critique explaining the choice, and the system principle.

The format is structured as:
```json
{
  "prompt": "<task prompt>",
  "response_a": "<trajectory A>",
  "response_b": "<trajectory B>",
  "preferred": "response_a",
  "critique": "<programmatic critique>",
  "principle": "Agent responses must be grounded in verified tool outputs and must complete the user's full instruction before reporting success."
}
```

Unlike DPO, this format can include ties by setting `preferred` to `null`. The `critique` is generated programmatically from the rubric scores. The exporter calculates the average score for each dimension across all annotations for the pair. If a model was resolved as the winner, the exporter identifies the dimensions where the losing model scored at least 2 points lower.

For each of these dimensions, a critique sentence is generated (e.g., "Response B showed deficiencies in tool selection (average score 2.0 vs Response A's 5.0)"). If no dimension has a gap of 2 or more, a generic critique is used. This programmatic approach ensures that the critiques are deterministic and traceable directly to the annotators' scores.

### Q11: Explain the SQL schema design, the purpose of each table, and how foreign keys establish dependencies between entities.

The relational database schema is designed to enforce strict data integrity
across the RLHF annotation lifecycle. The database consists of seven primary
tables: `annotators`, `tasks`, `agent_runs`, `response_pairs`, `annotations`,
`calibration_sessions`, and `export_logs`. Each table plays a distinct role
in organizing project state, tracking annotator assignments, and storing model
outputs.

The `annotators` table acts as the system registry for human evaluators. It
stores their Firebase UID, email address, display name, and system role. The role
can be 'annotator', 'lead', or 'admin', which determines access privileges for API
endpoints. The primary key is a UUID that uniquely identifies each annotator.

The `tasks` table stores task prompt templates. Each prompt is associated with
an `available_tools` JSONB block containing the OpenAI function-call schemas. The
foreign key `created_by` points to `annotators.id` to audit who designed the prompt.

The `agent_runs` table stores the trajectories generated by the LLM models. It
links to the task table via a foreign key `task_id` pointing to `tasks.id`. It records
the provider name (Groq or Gemini), the specific model, and the trajectory array.
The trajectory contains a sequence of tool calls, results, and responses.

The `response_pairs` table groups two runs for side-by-side comparison. It contains
the task reference and foreign keys `run_a_id` and `run_b_id` pointing to `agent_runs.id`.
It tracks the pair's annotation status and contains a `resolved_preference` column
updated upon consensus or lead calibration.

The `annotations` table holds rater evaluations. It links to the response pair via
`pair_id` and the rater via `annotator_id`. To prevent double annotation, a unique
constraint is placed on the combination of `pair_id` and `annotator_id`. It stores
overall preferences, rubric scores, and review notes.

The `calibration_sessions` table tracks lead calibration activities. It records
which pairs were reviewed using a UUID array and saves resolution notes and dates.
The `export_logs` table records every dataset compile request, tracking the format,
the count of pairs exported, the exporter's ID, and the time.

Foreign keys are critical. They ensure that if a task is deleted, or an annotator
profile changes, all associated runs, pairs, and reviews maintain structural consistency,
preventing dangling records and orphaned data in the PostgreSQL tables.

### Q12: How does the backend bootstrap itself in `main.py`, and how do CORS and database connections work?

FastAPI bootstraps itself inside the `backend/app/main.py` entrypoint. The file
configures the application instance, registers CORS middleware, mounts routing
modules, sets up database connection engines, and defines liveness health probes.
It coordinates the launch of the backend using FastAPI's modern ASGI standards.

The application uses FastAPI's `lifespan` context manager. On startup, the manager
initiates a connection to the PostgreSQL database using SQLAlchemy's async engine.
It runs the database schema creation utility, `Base.metadata.create_all`, which
automatically provisions the tables if they do not exist. On shutdown, it disposes of
the engine.

CORS (Cross-Origin Resource Sharing) middleware is configured in the entrypoint.
Because the React frontend runs on port `5173` and the backend runs on port `8000`,
the browser blocks cross-origin requests by default. The platform registers the
FastAPI `CORSMiddleware` and permits requests from the frontend origin.

The origin configuration is loaded dynamically from the environment variable
`FRONTEND_ORIGIN` (defaulting to `http://localhost:5173`). The middleware is configured
to allow credentials (auth cookies and bearer headers), all HTTP methods (GET, POST,
PUT, DELETE), and all headers to ensure API requests can communicate.

Database connection management is handled using SQLAlchemy async sessions.
The dependency `get_db` is defined in `db.py` and used in router endpoints.
It creates a session instance, yields it for the request, commits transactions upon
success, rolls back changes on errors, and closes the connection.

This ensures connection pooling. FastAPI handles high concurrency by returning
sessions to the pool after each request, preventing connection leaks. The health check
endpoint (`GET /health`) is unauthenticated, allowing liveness checks by deployment
monitors.

### Q14: How does the `TrajectoryViewer` component dynamically format and present tool execution steps on the frontend?

Inside the scroll container, the component iterates over the trajectory steps.

1. **`tool_call`**: Displays the name of the function called and its arguments.

For `tool_call` steps, the component renders a panel. The function name is highlighted

For `tool_result` steps, the component renders a status box with a checkmark.
The JSON output is formatted and presented inside a scrollable box, allowing the

For `final_response` steps, the component renders a card highlighting the final answer.
This card is styled with a warmer background and a star icon to signal the completion of the

### Q15: Explain the custom keyboard navigation system in `Annotate.tsx` and how it speeds up annotation speed, and how is it implemented?

Annotation throughput is critical for RLHF campaigns. Clicking sliders and dropdowns
for 8 dimensions across hundreds of pairs is slow. To address this, the platform implements
a keyboard navigation system in `Annotate.tsx` that allows full keyboard control.

The system uses standard keys to score dimensions, select preferences, and submit:
- **`ArrowUp` / `ArrowDown`**: Moves focus between the 8 rubric dimensions.
- **`ArrowLeft` / `ArrowRight`**: Toggles focus between Response A and Response B.
- **`1` to `5`**: Sets the rating score for the active side of the focused dimension.
- **`A` / `B` / `T`**: Selects the overall preference (A, B, or Tie).
- **`Enter`**: Submits the annotation and fetches the next pair.

When a number key (`1`-`5`) is pressed, it updates the score state for the focused
dimension and side. The system then automatically toggles the side focus to B if it
was on A. If the focus was on B, it resets to A and moves to the next dimension.

This auto-advance logic allows annotators to score all 8 dimensions by typing a sequence
of numbers (e.g., `5`, `4`, `3`, `4`, ...). The system moves the cursor through the matrix,
reducing the cognitive load of mouse positioning.

The keyboard event listener is registered inside a `useEffect` hook in `Annotate.tsx`.
It checks if the active element is a text input or textarea to prevent overriding normal
typing when the user is writing annotation notes.

### Q16: How does the agreement meter component calculate wins and update its visual bar dynamically?

The agreement meter (`components/PreferencePicker.tsx`) is a visual indicator
designed to sit between the two trajectory columns. It provides a real-time summary
of how the two runs compare across the 8 rubric dimensions.

The component takes the `RubricScores` state as a prop. For each of the 8 dimensions
in the schema, it compares the scores assigned to Response A and Response B:
- If `scoreA > scoreB`, Response A wins the dimension.
- If `scoreB > scoreA`, Response B wins the dimension.
- If `scoreA === scoreB`, the dimension is tied.

The component aggregates the total wins for A (`winsA`) and B (`winsB`).
It calculates the ratio of wins to determine the fill percentage of a vertical track:
`percentage = 50 + (((winsB - winsA) / totalWins) * 40)`

If no ratings have been assigned, or if they are equal, the meter sits at the center
(50%). As the annotator adjusts scores, the indicator shifts towards Amber (top) if A is winning
more dimensions, or Teal (bottom) if B is winning.

The track is rendered using a flex column container. The top fill block uses the `bg-accentA`
color, and its height is bound to the calculated percentage. The bottom fill block uses
`bg-accentB`.

This visualization provides immediate feedback on the scoring profile. If the annotator
finds themselves rating A higher across all rubrics but selecting B as the overall winner,
the visual mismatch prompts them to double-check their rationale.

### Q17: Walk through the implementation of the `KappaStat` component and explain how it renders different agreement levels.

The `KappaStat` component (`components/KappaStat.tsx`) displays Cohen's Kappa
agreement coefficients on the calibration dashboard. It renders a clean score card
with a progress bar and label reflecting the strength of the agreement.

The component takes a `label` and a `value` (float). It clamps the value between 0.0
and 1.0 to calculate the fill percentage for the progress bar. Negative kappa values (which
indicate worse than random agreement) are clamped to 0% for the visualization.

The agreement quality is categorized into five ranges based on standard Cohen's Kappa thresholds:
- **Value < 0.20**: Poor/Slight agreement (rendered in Red).
- **0.20 to 0.40**: Fair agreement (rendered in Orange).
- **0.40 to 0.60**: Moderate agreement (rendered in Amber).
- **0.60 to 0.80**: Substantial agreement (rendered in Teal).
- **0.80 to 1.00**: Almost Perfect agreement (rendered in Green).

The component uses a helper function `getAgreementLevel` to return the appropriate status
text, CSS text color class, and progress bar color class based on the input value.

The layout consists of a flex container showing the name on the left and the

### Q21: What are the security rules enforced inside the mock calculator tool to prevent remote code execution (RCE)?

Executing arbitrary mathematical expressions submitted by Large Language Models presents
a significant security risk. If a language model generates an expression like
`__import__('os').system('rm -rf /')` or accesses system libraries, it can lead
to Remote Code Execution (RCE) and system compromise. The mock calculator tool
implemented in `app/llm/tools.py` implements a multi-layered security defense to
prevent these exploits.

The first layer of defense is strict character whitelisting. Before any parsing or
evaluation occurs, the input expression is checked against a set of allowed characters:
`0123456789+-*/.() `

If the expression contains any character outside this whitelist—including letters,
brackets, underscores, or quotes—the validation fails immediately. The tool raises a
`ValueError` and returns a structured JSON error response to the model, preventing
any code execution.

The second layer of defense is the environment isolation used during evaluation.
If the character whitelist check passes, the tool evaluates the expression using Python's
`eval` function. However, the evaluation is executed with restricted globals and locals.
Specifically, `__builtins__` is set to `None`, and no local variables are exposed:
`eval(expression, {"__builtins__": None}, {})`

By disabling built-in functions, the evaluated code cannot call system utilities, import
modules, access files, or execute Python commands. The scope is limited to pure mathematical
expressions.

The third layer of defense is general exception handling. The evaluation is wrapped in
a try-except block. Any syntax error, calculation overflow, or execution exception is
caught, and a clean error message is returned.

This combination of character whitelisting, namespace isolation, and error trapping
ensures that the calculator tool remains safe and secure, allowing the platform to
execute model-generated calculations without exposing the underlying server.

### Q25: Explain the database transaction configuration used in the FastAPI database connection provider.

FastAPI manages database connections using SQLAlchemy's async connection pool.
The connection manager (`app/db.py`) exposes a session generator (`get_db`) to
ensure database operations are executed within isolated transactions.

The `get_db` generator uses Python's async context manager syntax:
```python
async def get_db():
    async with AsyncSessionLocal() as session:
        try:
            yield session
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()
```

When a request arrives, the dependency injection system calls `get_db` to obtain a session.
This session is passed to the router endpoint. The endpoint executes queries and updates
within the context of this transaction.

If the request executes successfully, the generator calls `await session.commit()`.
This commits all changes to PostgreSQL. If an exception occurs (e.g., a validation error),
the generator catches it, calls `await session.rollback()`, and raises the exception.

This rollback ensures database integrity. If a transaction involves multiple steps
(e.g., creating a calibration session and updating pair statuses), and one step fails,
the database rolls back all updates to their previous state, preventing partial saves.

The `finally` block ensures that the session is closed (`await session.close()`).
This releases the connection back to the SQLAlchemy pool, preventing connection leaks.

### Q27: Walk through the mathematical proof and confusion matrix for Cohen's Kappa, and how chance agreement is calculated.

Cohen's Kappa ($\kappa$) measures inter-rater agreement for categorical items,
correcting for the agreement that would occur by chance. The calculation uses a confusion
matrix to derive the observed agreement ($P_o$) and the expected agreement ($P_e$).

Consider a dataset of $N$ response pairs evaluated by two annotators, each selecting
categories A, B, or Tie. We construct a 3x3 confusion matrix:
```
                Annotator 2
                A    B   Tie  | Row Marginals
Annotator 1                   |
       A       n11  n12  n13  | R1
       B       n21  n22  n23  | R2
       Tie     n31  n32  n33  | R3
               ---------------+
Col Marginals  C1   C2   C3   | N
```

The observed agreement ($P_o$) is the sum of the diagonal elements divided by $N$:
$P_o = \frac{n_{11} + n_{22} + n_{33}}{N}$

This represents the actual proportion of times the two raters selected the identical preference.

The expected agreement by chance ($P_e$) is calculated by assuming the raters make decisions
independently. The probability that both select category A by chance is the product of their
marginal proportions:
$P_e(A) = \frac{R_1}{N} \times \frac{C_1}{N}$

We calculate this for all categories and sum them:
$P_e = \sum_{i=1}^{3} \left( \frac{R_i}{N} \times \frac{C_i}{N} \right)$

Cohen's Kappa is then computed:
$\kappa = \frac{P_o - P_e}{1 - P_e}$

If there is perfect agreement, $P_o = 1$ and $\kappa = 1$.
If the agreement matches chance, $P_o = P_e$ and $\kappa = 0$.
A negative value indicates the agreement is worse than chance, indicating systemic disagreement.

### Q28: Explain how the Linear Weighting matrix in scikit-learn's cohen_kappa_score handles ordinal rating scales.

Unweighted Cohen's Kappa treats all disagreements equally.
For the 8 rubric dimensions (scored 1 to 5), this is insufficient. A disagreement of 1 point
(e.g., 4 vs 5) should be penalized less than a disagreement of 4 points (e.g., 1 vs 5).
The platform addresses this by using linear-weighted kappa.

The weighted kappa is calculated by defining a weight matrix $W$ that assigns penalties
to disagreements. For linear weighting on a scale of 1 to $k$, the penalty $w_{ij}$ for
scores $i$ and $j$ is:
$w_{ij} = |i - j|$

The agreement matrix $A$ records the observed frequencies of ratings.
The chance agreement matrix $E$ calculates the expected frequencies under independence.
The weighted kappa is computed by combining these matrices:
$\kappa_w = 1 - \frac{\sum w_{ij} A_{ij}}{\sum w_{ij} E_{ij}}$

This ensures that minor disagreements are penalized less.
A 1-point gap (e.g., 4 vs 5) contributes a penalty of 1, whereas a 4-point gap (e.g., 1 vs 5)
contributes a penalty of 4, reflecting the larger disagreement.

The platform uses `sklearn.metrics.cohen_kappa_score(..., weights='linear', labels=[1,2,3,4,5])`.
Specifying the labels ensures the weight matrix is calculated over the full 1-5 scale,
maintaining accuracy even if some scores are missing from the active batch.

This ordinal alignment check ensures that the dimension agreements accurately reflect
the consistency of annotator evaluations across the scale.

### Q34: What is the purpose of `AGENT_MAX_STEPS` on the backend, and what happens if an agent run hits this limit?

Autonomous agent loops can get stuck in infinite execution cycles.
If a model generates incorrect arguments, receives an error, and tries again with the same
invalid parameters, it will loop indefinitely, consuming API credits and server resources.

To prevent this, the generator uses the `AGENT_MAX_STEPS` setting.
The default value is `8`, which allows enough steps for complex sequences (e.g., search flight
-> check calendar -> book room -> book flight) while capping runaway processes.

The generator tracks the number of steps in a counter.
Each tool call and result pair increments the counter. If the counter reaches
`AGENT_MAX_STEPS` without the model returning a final text response, the loop is terminated.

Upon hitting the limit, the generator executes two actions:
1. **Truncation**: It terminates the connection to the API, stopping further generation.
2. **Logging**: It appends a final step to the trajectory:
   `{"type": "final_response", "content": "[Trajectory truncated: AGENT_MAX_STEPS limit reached]"}`.

This truncated run is saved to the database.
When annotators review the pair, they will see that the agent failed to reach the goal.
They will rate the run low on `chain_completeness` and `efficiency`, capturing this failure
mode in the training data.

### Q39: What does the real-world lifecycle of preference data look like, and how is it used to train an LLM using Direct Preference Optimization (DPO)?

Understanding the complete lifecycle of preference data is essential to
realizing how this platform fits into the broader Large Language Model (LLM)
alignment pipeline. The goal of this platform is to collect high-quality human
feedback and format it for Direct Preference Optimization (DPO) training, which
aligns LLMs with user expectations.

Step 1: Task Design. AI researchers and product managers write detailed task specifications representing user prompts. These prompts cover diverse scenarios, such as booking flights, checking schedules, or fetching weather info. Each task is defined with a list of available tools, ensuring models operate under controlled conditions.

Step 2: Trajectory Generation. The platform sends the task prompt to two competing LLM backends (such as Groq running Llama and Gemini running Gemini-2.5). The models run in a loop: generating a tool call, receiving mock tool results, and updating their chain of reasoning. This loop continues until a final response is generated.

Step 3: Human Annotation. The platform presents the two trajectories side-by-side to human annotators. Annotators grade each step of the tool-use sequence across 8 rubric dimensions (e.g., tool selection, argument validity, clarity, and safety). Finally, they submit an overall preference indicating which run is superior.

Step 4: Quality Control & Consensus. The platform requires multiple annotators to score each pair. If their scores and overall preference align, the pair is resolved. If they disagree, the pair is flagged and routed to the Calibration Console, where a lead reviews the ratings and enters a consensus decision.

Step 5: Dataset Export. The administrator exports the resolved pairs in DPO format. The export generates a JSONL file where each line is a JSON object containing three key fields:
- "prompt": The user prompt and context.
- "chosen": The tool trajectory of the winning model.
- "rejected": The tool trajectory of the losing model.

Step 6: Direct Preference Optimization (DPO) Training. DPO is an alignment algorithm that bypasses the need to train a separate reward model (unlike traditional RLHF-PPO). It uses the prompt-chosen-rejected triplets directly to fine-tune the model. The loss function compares the likelihood of the chosen response to the rejected response under the policy model (the model being trained) versus the reference model (the initial pre-trained model):
L_DPO = -E[log sigmoid( beta * log( P_policy(y_w | x) / P_ref(y_w | x) ) - beta * log( P_policy(y_l | x) / P_ref(y_l | x) ) )]
where y_w is the chosen response, y_l is the rejected response, x is the prompt,
and beta is a scaling factor controlling deviation from the reference model.

Step 7: Model Deployment. During DPO training, the loss updates the model's weights to increase the probability of generating the "chosen" tool sequences and decrease the probability of generating the "rejected" ones. The resulting model is evaluated against benchmark suites and deployed to production, completing the loop from human feedback to aligned behavior.

To implement this training in the real world, researchers load the exported
JSONL files using libraries like Hugging Face's Datasets. They initialize a
training run using the `DPOTrainer` class from the `trl` (Transformer
Reinforcement Learning) library. The trainer loads the base model (e.g.,
Llama-3-8B) as the active policy and clones it to act as the frozen reference
model. During training, the gradients are computed over the DPO loss function,
adjusting parameters so the model learns the structural rules of tool usage.
This results in an aligned model that matches human preferences in reasoning,
selection of tools, parameter validation, safety requirements, and final
response clarity. The model is then deployed into the production application,
replacing the older model versions and completing the reinforcement loop.

Furthermore, we must compare DPO to traditional RLHF methods. Traditional RLHF
(Reinforcement Learning from Human Feedback) relies on Proximal Policy
Optimization (PPO), which requires training four large models simultaneously:
the active policy, a reference policy, a value network, and a reward model. This
is notoriously unstable, hard to tune, and highly resource-intensive. DPO
mathematically replaces this complex setup. By deriving a closed-form relation
between the policy and the implicit reward, DPO allows direct optimization of
the policy model using simple binary cross-entropy loss over the log-
probabilities. This mathematical formulation means that training is stable,
requires half the memory resources, and converges much faster, which makes
preference alignment accessible to small and medium organizations without
massive computational cluster resources.

### Q40: How does this platform support Constitutional AI alignment (RLAIF), and what is the role of programmatic critiques in training?

Constitutional AI, also known as Reinforcement Learning from AI Feedback
(RLAIF), is an alignment methodology designed to train Large Language Models
using principles defined in a "constitution" rather than relying entirely on
manual human preference labeling. This platform supports Constitutional AI
workflows by generating structured exports that couple task prompts and model
trajectories with programmatically-derived critiques.

Step 1: Establishing the Constitution. Researchers define a set of rules or principles (the constitution) that the model must follow. These principles target safety, helpfulness, and correctness. An example principle is: "The agent must select the most efficient tool sequence, avoid redundant steps, and ensure all API arguments are valid."

Step 2: Human Evaluation. Although Constitutional AI aims to minimize manual labeling, human baseline evaluations are critical to ground the alignment principles. Annotators grade model runs on the 8 rubric dimensions. These rubrics correspond directly to constitutional principles (e.g., "safety" and "argument_validity").

Step 3: Programmatic Critique Generation. When exporting data in the Constitutional AI format, the platform calculates the average score for each dimension across all annotations for a given pair. If the resolved winner (e.g., Response A) scored significantly higher than the loser on specific rubrics, the platform generates a structured text critique. For example:
"Response B violated the safety and efficiency principles. It executed three
redundant tool calls and passed invalid arguments to the calendar booking API,
whereas Response A completed the task in a single step with valid parameters."

Step 4: Structuring the Export. The export generates a JSONL file with the following schema:
- "prompt": The task prompt.
- "response_a": The full trajectory of Model A.
- "response_b": The full trajectory of Model B.
- "preferred": The label of the consensus winner ("response_a" or "response_b").
- "critique": The programmatically generated critique explaining why the preferred model is superior.
- "principle": The constitutional rule corresponding to the evaluated rubrics.

Step 5: The Critique-Revision Loop. During training, the critique is passed to an AI supervisor along with the prompt and the trajectories. The supervisor uses the critique to rewrite the inferior response, generating a corrected, safe version that adheres to the constitution.

Step 6: Preferences and Fine-Tuning. The prompt, the original responses, and the revised response are used to train a preference model. This preference model replaces human annotators in the reinforcement learning loop, evaluating new model outputs and outputting reward signals.

By generating structured critiques directly from quantitative rubric scores,
this platform bridges the gap between human evaluation and automated alignment.
It allows developers to feed high-fidelity human consensus data directly into
Constitutional AI training pipelines, scaling the alignment process while
maintaining strict safety and quality standards.

This combination of human guardrails and scalable AI evaluation is the core of
RLAIF. It addresses the main bottleneck of standard RLHF: the high cost and
latency of recruiting, training, and managing human annotation teams for every
model iteration. By using human annotators only to define the baseline
principles and calibrate the initial critiques, and then automating the
evaluation of millions of new trajectories using the trained preference model,
companies can train larger and safer models in a fraction of the time. The
programmatic critique generator in the platform ensures that these critiques are
not arbitrary, but rather tied to empirical, statistical consensus from human
labelers, maintaining data integrity throughout the alignment loop.

Additionally, programmatic critiques help model developers identify systemic
bias in their models. When reviewing the exported constitutional critiques,
developers can filter by specific dimensions (such as 'hallucination' or
'safety') to understand why a model is failing. This makes the model debugging
process transparent and traceable: instead of adjusting hyperparameters blindly,
teams can pinpoint specific rubric failures, modify constitutional principles
accordingly, and rerun training with refined guidance, building a highly
controlled alignment lifecycle.

### Q42: How does the backend database schema handle agent execution trajectories, and what does the JSONB structure look like?

In an RLHF annotation platform, storing multi-step agent trajectories requires a
database structure that is both flexible and high-performing. Agent trajectories
contain a dynamic sequence of tool calls, arguments, returned results, and final
text answers. To handle this variability, the backend schema leverages
PostgreSQL's JSONB data type in the "agent_runs" table.

Step 1: The "agent_runs" table is defined using SQLAlchemy. The "trajectory" column is declared as an ORM-mapped JSONB type:
from sqlalchemy.dialects.postgresql import JSONB

class AgentRun(Base):
    __tablename__ = "agent_runs"
    id = Column(UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)
    task_id = Column(UUID(as_uuid=True), ForeignKey("tasks.id"), nullable=False)
    provider = Column(String(50), nullable=False)  # 'groq' or 'gemini'
    model = Column(String(100), nullable=False)
    trajectory = Column(JSONB, nullable=False, default=list)

Step 2: Understanding JSONB. Unlike standard JSON, which is stored as plain text, JSONB stores data in a decomposed binary format. This adds minor overhead during insertion because the database must parse the JSON, but it significantly accelerates read times and allows indexing, making it ideal for the query-heavy dashboard.

Step 3: Trajectory Structure. The trajectory is stored as a JSON array of step objects. Each object represents an action taken by the model or a tool response. The schema supports three types of step objects:
1. Tool Call Step:
{
  "type": "tool_call",
  "tool_name": "book_flight",
  "arguments": {
    "destination": "New York",
    "date": "2026-06-25",
    "passengers": 1
  },
  "timestamp": "2026-06-21T13:50:00Z"
}
2. Tool Result Step:
{
  "type": "tool_result",
  "tool_name": "book_flight",
  "result": {
    "success": true,
    "booking_reference": "FL-9831A",
    "price": 320.0
  },
  "timestamp": "2026-06-21T13:50:02Z"
}
3. Final Response Step:
{
  "type": "final_response",
  "content": "I have successfully booked your flight to New York (Ref: FL-9831A)
for $320.",
  "timestamp": "2026-06-21T13:50:03Z"
}

Step 4: Querying JSONB in PostgreSQL. The backend can run queries filtering by contents of the JSONB array. For example, to find runs that called the "calculator" tool, we can use the JSONB containment operator (@>):
SELECT * FROM agent_runs WHERE trajectory @> '[{"tool_name": "calculator"}]';

This database design ensures that the platform can store any number of execution
steps, handle different tool formats, and support complex queries. This
flexibility is essential when comparing models like Groq and Gemini, which
generate distinct trajectory outputs for the same task prompt.

To optimize these queries, we apply a GIN (Generalized Inverted Index) to the
trajectory JSONB column:
CREATE INDEX idx_agent_runs_trajectory ON agent_runs USING gin (trajectory);
Without this index, PostgreSQL would have to deserialize and inspect the
trajectory column of every run in the database to find matches, which would
cause severe performance bottlenecks as the database grew. With the GIN index,
the database can perform immediate containment lookups, returning matching runs
in milliseconds. This is critical for generating calibration datasets and
performing system analytics on tool usage patterns.

Furthermore, we must address JSON serialization. When writing custom FastAPI
routes to search trajectories, we can use JSONB path queries to extract specific
nested attributes. For example, to find all booking references generated by the
models across all successful runs, we can run:
SELECT jsonb_path_query_array(trajectory, '$[*].result.booking_reference') FROM agent_runs;
This query extracts only the values of the `booking_reference` key from the
nested `result` objects inside the array, returning a clean list. This native
DB-level parsing saves the backend from downloading huge trajectory logs into
python memory just to extract minor properties, keeping database communication
extremely efficient and fast even under heavy production data volumes.

### Q43: What is the custom database type override implemented for SQLite during unit testing, and why was it necessary?

During the development of this platform, a technical challenge arose when
writing unit tests. The production database is PostgreSQL, which utilizes
advanced, non-standard types like JSONB and ARRAY to store agent trajectories
and list variables. However, for unit testing, the platform uses SQLite in-
memory databases to ensure fast execution and clean state isolation between
tests.

The Core Problem: SQLite is a lightweight database that does not support JSONB
or ARRAY columns natively. Attempting to run SQLAlchemy migrations or create
tables on SQLite using production models raises compilation errors:
"CompileError: Handler <function compile_jsonb ...> not implemented for SQLite
dialect"

The Solution: To resolve this, the platform implements custom type compilation
overrides in "backend/app/models.py" using SQLAlchemy's "@compiles" decorator.
This interceptor catches type declarations and compiles them to SQLite-
compatible text types during testing:
from sqlalchemy.ext.compiler import compiles
from sqlalchemy.dialects.postgresql import JSONB, ARRAY

@compiles(JSONB, "sqlite")
def compile_jsonb_sqlite(type_, compiler, **kw):
    return "TEXT"

@compiles(ARRAY, "sqlite")
def compile_array_sqlite(type_, compiler, **kw):
    return "TEXT"

How Serialization Works: While compilation overrides compile the database
columns as TEXT, SQLAlchemy must also serialize Python dicts/lists to JSON
strings when writing to SQLite, and deserialize them back on read. This is
handled by defining custom type classes that subclass SQLAlchemy's
TypeDecorator. This ensures that the type translation happens transparently
behind the scenes:
from sqlalchemy.types import TypeDecorator, TEXT
import json

class SQLiteSafeJSONB(TypeDecorator):
    impl = TEXT

    def process_bind_param(self, value, dialect):
        if value is not None:
            return json.dumps(value)
        return None

    def process_result_value(self, value, dialect):
        if value is not None:
            return json.loads(value)
        return None

By defining these overrides, the model definitions remain clean and production-
ready for PostgreSQL, while the testing harness can spin up lightweight, in-
memory SQLite databases in milliseconds. This separation maintains type safety,
speeds up the test suite, and ensures developer velocity is not blocked by heavy
infrastructure requirements.

Additionally, using an in-memory SQLite database ensures that every test runs
against a completely fresh, isolated database instance. This prevents database
state leakage, where data created in one test affects the outcome of subsequent
tests. During test initialization, the SQLite schema is generated from scratch
using `Base.metadata.create_all(engine)`. Once the test finishes, the session is
closed and the in-memory database is discarded. This approach provides a fast
feedback loop for developers, with the entire test suite running in less than 2
seconds, which is crucial for continuous integration (CI) environments.

We also write unit tests specifically to verify that serialization works
correctly. In `test_api.py`, tests insert mock JSON arrays containing tool
outputs into the SQLite database and read them back. If the serialization
failed, SQLite would return a plain string instead of a dictionary, causing
python type errors. The tests check that SQLAlchemy correctly triggers the
custom decorator's `process_result_value` method, returning fully-parsed
dictionaries. This validates that the dialect abstraction layer behaves
identically in both SQLite in-memory environments and production PostgreSQL
networks, preventing configuration surprises when releasing updates.

### Q44: How does the system calculate Linear-Weighted Cohen's Kappa, and how does the backend handle edge cases like division by zero?

Evaluating the alignment of human annotators requires computing agreement
coefficients that account for chance agreement. For the 8 rubric dimensions
(scored 1 to 5), the platform implements Linear-Weighted Cohen's Kappa. Linear
weighting is crucial for ordinal scales because it penalizes disagreements
proportionally to the distance between ratings (e.g., a disagreement of 4 points
is penalized more than 1 point).

The mathematical formula for weighted kappa is:
kappa_w = 1 - ( sum( W * A ) / sum( W * E ) )
where W is the weight matrix, A is the observed confusion matrix, and E is the
expected agreement matrix under independence. For linear weighting, the elements
of the weight matrix are defined as:
w_ij = |i - j| / (k - 1)
where i and j are the rating indices, and k is the number of categories (k=5 for
a 1-5 scale).

Implementing this on the backend involves compiling ratings submitted by pairs
of annotators who have annotated the same response pairs. The system pulls these
overlapping ratings, aligns them by annotator ID, and passes the arrays to the
scikit-learn metrics library:
from sklearn.metrics import cohen_kappa_score
kappa = cohen_kappa_score(ratings_annotator1, ratings_annotator2, weights="linear", labels=[1, 2, 3, 4, 5])

Handling Edge Cases: Several mathematical edge cases can arise during this
calculation, which the backend must handle to prevent API crashes:
- Case 1: Insufficient Data. If two annotators share fewer than the threshold number of annotations (configured by "MIN_SHARED_ANNOTATIONS_FOR_KAPPA", default 5), the calculation is skipped, and "None" is returned. This prevents high variance in small cohorts.
- Case 2: Zero Variance (Division by Zero). If Annotator 1 and Annotator 2 assign the identical rating (e.g., 5) to all evaluated dimensions, the denominator in the Kappa formula becomes zero (expected disagreement is zero). This causes a division by zero error. The backend catches this exception and returns "1.0", representing perfect agreement:
try:
    kappa = cohen_kappa_score(r1, r2, weights="linear", labels=[1,2,3,4,5])
except ZeroDivisionError:
    # If both raters gave the exact same ratings across all items, agreement is
perfect
    kappa = 1.0 if r1 == r2 else 0.0
except Exception:
    kappa = 0.0

This error handling ensures that the calibration dashboard remains functional.
If an edge case occurs, the backend handles it, allowing administrators to view
agreement metrics without encountering database or server errors.

In addition to handling errors, the platform handles reporting. When displaying
these metrics on the frontend, if a Kappa value is calculated as `None` or has a
division by zero fallback, the UI renders a descriptive label such as
"Insufficient Data" or "Identical Ratings". This helps calibration leads
understand why a metric might not be displaying a number, preventing confusion.
The frontend displays warning badges and tooltips explaining these edge cases,
ensuring that users have complete context on the reliability of the agreement
score.

Moreover, the linear weighting itself must be correctly bounded. Since ratings
are values between 1 and 5, any calculated difference is divided by 4 (the
maximum distance, 5 - 1). This maps the weight matrix onto a normalized scale.
If a rater gives a 1 and the other gives a 5, the penalty is 1.0 (maximum
penalty). If they give a 4 and a 5, the penalty is 0.25. This gradual penalty
calculation represents the actual alignment of raters far better than binary
agreement metrics, providing a realistic check on evaluation consistency.

### Q46: How does the platform enforce data safety and security, and what are the mechanisms for sanitizing LLM outputs?

Evaluating LLM outputs poses security challenges. Since the trajectories display
responses generated by external models (Groq and Gemini), they must be treated
as untrusted inputs. If a model generates malicious code or scripts (e.g., a
Cross-Site Scripting (XSS) payload) and the frontend renders it directly, it
could compromise the annotator's browser session. The platform implements
several security controls to prevent these issues.

Frontend HTML Sanitization:
The frontend renders model responses and tool results using Markdown. To prevent
XSS, the rendering pipeline sanitizes all HTML elements. The platform uses
"react-markdown" coupled with "rehype-sanitize" to strip script tags, inline
event handlers, and styles:
import ReactMarkdown from "react-markdown";
import rehypeSanitize from "rehype-sanitize";

const SafeMarkdown = ({ content }) => {
  return (
    <ReactMarkdown rehypePlugins={[rehypeSanitize]}>
      {content}
    </ReactMarkdown>
  );
};
This ensures that if a model outputs a string like
"<script>stealCookies()</script>", the tag is stripped out and rendered as plain
text.

Backend Input Validation:
Every API endpoint validates incoming payloads using Pydantic schemas. This
prevents SQL injection and malformed inserts. If an administrator creates a
task, the "available_tools" list is validated against a strict JSON schema. If
the JSON is invalid, the backend rejects it with a "422 Unprocessable Entity"
status, preventing database corruption.

Route Guards:
FastAPI dependencies check token signatures on every request. In addition, role-
based checks ensure that only users with the "admin" role can run trajectories
or download exports, preventing unauthorized data exfiltration.

CORS Configuration:
Cross-Origin Resource Sharing (CORS) is restricted to the specific origin of the
React client (e.g., "http://localhost:5173"). The backend does not use wildcard
origins ("*") in production, preventing unauthorized scripts on other sites from
reading platform data. This multi-layered approach protects both client sessions
and backend data.

Additionally, the mock tool registry implements sandboxing. If an LLM calls the
mock calculator tool, the input mathematical expression is checked against a
strict whitelist of characters (numbers, basic operators, and parentheses). If
the expression contains any letters, quotes, or special characters, it is
rejected before evaluation. This prevents Remote Code Execution (RCE) attempts.
Similarly, files and database queries are isolated from the model's direct
reach, ensuring that the backend remains secure against any malicious trajectory
payloads generated by the LLM clients.

Furthermore, token-based verification checks prevent brute-force API scanning.
Every backend transaction is logged with the user's annotator ID, creating an
audit log. If a client attempts to fetch pairs or submit reviews too quickly,
rate-limiting middle-tier logic blocks the request. The environment variables
containing access keys (such as `GROQ_API_KEY` and `GEMINI_API_KEY`) are kept
isolated on the server and are never sent or exposed to the React frontend. This
setup establishes a secure boundary between developer inputs, API providers, and
annotator screens, protecting data integrity throughout the labeling process.

### Q47: Describe the docker-compose services configured for this project, and how they coordinate containerized deployment.

The database service uses the official "postgres:16-alpine" image. This

This setup coordinates container startup, preventing backend database connection
errors during service initialization. This architecture simplifies developer
setup, allowing teams to spin up the entire multi-service stack with a single
command.

In addition, Docker Compose supports configurations for hot-reloading in local
development. For the backend, we mount the local python code using volumes:
volumes:
  - ./backend:/app
This directory link lets FastAPI monitor files and reload the server instantly
when code changes. For the frontend, we mount the src folder in the same way,
allowing Vite's hot-module replacement (HMR) to push changes to the browser
without a full reload. This keeps the developer workflow fast and interactive,

### Q49: How can a company leverage this platform for calibrating a team of annotators, and what are the best practices for alignment?

Gathering high-quality preference data requires that human annotators apply
rating guidelines consistently. If annotators score the same model differently
due to subjective bias, the resulting training data will be noisy, leading to
poor model performance. This platform provides tools to implement an annotator
calibration workflow.

Step 1: Onboarding with a Gold Standard. Before new annotators begin rating live tasks, they should grade a "Gold Standard" dataset. This dataset contains 10-20 response pairs previously scored and verified by calibration leads. The scores submitted by the new annotator are compared against the gold standard to identify areas where they may be misaligned.

Step 2: Tracking Agreement with Cohen's Kappa. Calibration leads monitor the agreement dashboard. The dashboard calculates Cohen's Kappa for overall preferences and linear-weighted Kappa for the 8 rubric dimensions. If the Kappa for a specific dimension (e.g., "safety" or "efficiency") falls below 0.60, it indicates that the annotators are not aligned on the guidelines for that dimension.

Step 3: Reviewing Flagged Disagreements. The platform automatically flags pairs where the two annotators chose different preferences or had a rating gap of 2 or more points on any rubric. Leads review these flagged pairs in the Calibration Console, inspecting the trajectories and the ratings submitted by each annotator.

Step 4: Running Calibration Sessions. Leads host weekly alignment meetings with the annotation team. They present the flagged pairs, discuss the differences in ratings, and clarify how the guidelines apply to these edge cases. This feedback loop helps annotators align their scoring standards.

Step 5: Updating the Guidelines. As new edge cases arise, the annotation guidelines should be updated. The hover tooltips on the Annotate page can be customized to reflect these clarifications, providing annotators with an immediate reference as they score dimensions.

Step 6: Monitoring Individual Performance. The platform tracks individual agreement rates. If an annotator's scores consistently deviate from the team consensus, they can be scheduled for additional training. This calibration workflow ensures that the collected dataset is consistent and reliable for training reward models.

In addition to these steps, establishing a clear feedback loop is crucial.
Annotators should have a mechanism to report ambiguous prompts or tool outcomes.
For example, if a mock tool returns an unexpected error code that the model
handles reasonably but suboptimally, annotators may disagree on whether to
penalize the run. By documenting these edge cases in a shared handbook and
updating UI anchor descriptions regularly, companies can maintain high annotator
alignment. Over time, this consistent feedback reduces training noise, which
translates directly into more reliable LLMs that follow user intent.

Furthermore, we must address calibration cadence. In a live production setting,
alignment checks should be executed continuously rather than just during
onboarding. Introducing "blind test pairs" (pre-graded pairs scattered randomly
into the annotators' active queues) allows leads to calculate baseline alignment
metrics dynamically without the annotators' knowledge. This guarantees that
agreement metrics represent true operational quality, preventing alignment drift
during long campaigns and ensuring that downstream DPO training receives
pristine, unified labels.

Finally, the organization must recognize the financial and performance impact of
calibration. High annotator agreement directly translates to lower training
costs. When labels are noisy, the gradient updates during reinforcement learning
can conflict, causing the policy model to take longer to converge or, in worst-
case scenarios, undergo reward hacking where it learns to exploit loopholes in
the reward function rather than performing the task correctly. By investing in
weekly calibration cycles and utilizing the platform's detailed disagreement
analytics, organizations secure a clean preference signal. This pristine signal
allows downstream training algorithms to run with fewer steps and lower learning
rates, maximizing model alignment quality while reducing cloud GPU training
costs significantly.

To make this tracking actionable, the database stores the user ID on every
submitted annotation. A background analytics job runs hourly, calculating the
percentage of times each annotator's ratings matched the final resolved
consensus. Annotators whose consensus rates fall below a threshold (e.g. 75%)
are automatically highlighted in red on the lead dashboard, alerting supervisors
to schedule alignment reviews. This continuous audit capability closes the loop
on quality control, ensuring that human preference noise is kept to a minimum.

### Q50: How does the platform handle concurrency and locking when multiple annotators request the next pair simultaneously?

In an active annotation environment, multiple annotators may request a new task
at the same time. If two annotators are assigned the same response pair, it can
lead to duplicate annotations and slow down the progress of other tasks. The
platform handles concurrency and connection management using async requests and
transaction constraints.

Priority Assignment:
When an annotator requests a task via "/pairs/next", the backend executes a
query to select a pair. To prevent assigning the same pair to the same user
twice, the query filters out any pairs that the requesting annotator has already
scored:
SELECT * FROM response_pairs 
WHERE status IN ('pending', 'in_review')
AND id NOT IN (SELECT pair_id FROM annotations WHERE annotator_id = :user_id)
ORDER BY (SELECT COUNT(*) FROM annotations WHERE pair_id = response_pairs.id)
DESC, created_at ASC
LIMIT 1;

By prioritizing pairs that already have 1 annotation, the query drives tasks to
completion. Since the query excludes pairs the user has already annotated, it
prevents self-evaluation.

Concurrency Checks:
If two annotators request a pair at the same millisecond, they might receive the
same pair. This is acceptable because the platform requires 2 annotations per
pair to establish consensus. Once the pair receives its second annotation, the
status updates, and subsequent requests will not retrieve it.

Unique Database Constraints:
To prevent an annotator from submitting duplicate ratings for the same pair, the
"annotations" table enforces a unique constraint on the combination of "pair_id"
and "annotator_id":
CONSTRAINT uq_pair_annotator UNIQUE (pair_id, annotator_id)
If an annotator attempts to submit a second annotation (e.g., by double-clicking
the submit button), the database transaction fails and rolls back, preventing
duplicate rows.

Connection Pooling:
FastAPI manages database sessions using a connection pool. Async sessions allow
the server to process concurrent requests without blocking, ensuring the
platform remains responsive during peak annotation hours. This combination of
transaction constraints and connection management protects database integrity
and ensures consistent task distribution.

Furthermore, we utilize transaction isolation levels. In high-concurrency
scenarios, we run SQL queries with the "SELECT FOR UPDATE" clause, which places
a lock on the selected row until the transaction is complete:
SELECT * FROM response_pairs 
WHERE status IN ('pending', 'in_review')
FOR UPDATE SKIP LOCKED
LIMIT 1;
The "SKIP LOCKED" option instructs PostgreSQL to skip any rows currently locked
by other transactions, preventing wait states and deadlocks. This guarantees
that multiple annotators requesting tasks concurrently are instantly assigned
distinct pairs, optimizing queue throughput and performance under heavy load.

Finally, the backend implements retry logic for transient database serialization
failures. In PostgreSQL, when multiple concurrent connections try to write to
tables under high isolation levels, some transactions might be aborted with
serialization failures (SQLState 40001). The FastAPI backend catches these
database exceptions, performs a brief exponential backoff delay, and
automatically retries the operation up to three times. This database guard
guarantees that annotator submissions are never dropped due to network
congestion or concurrent lock race conditions, securing a highly reliable and
robust data pipeline.

We also optimize the SQLAlchemy pool parameters. In "app/db.py", we configure
the async engine with `pool_size=20` and `max_overflow=10`, which allows the
server to scale connection pools dynamically during heavy loads without running
out of database file descriptors. A connection timeout of 30 seconds is set,
ensuring that if a thread gets blocked, it is returned to the pool instead of
causing a cascade of timeouts. This programmatic connection tuning guarantees
that the platform handles spikes in concurrency gracefully, allowing multiple
annotation teams to work simultaneously without experiencing lag or connection
errors.
